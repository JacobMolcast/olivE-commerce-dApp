Objectives

Design and develop a decentralized web application built on top of blockchain infrastructure that bring closer first providers with end clients. A public website will present the world the ethic, work and product of a community. A user will be able to navigate, open and read different sections and purchase the product if a Metamask extension with money is installed in the browser. After a purchase, a relationship is created, being final clients the ones that vote with the support of money for the success of the project. A non-shareable token will document this transaction and serve as a receipt for the product to be sent. While the product is on transit, it needs to change hands several times. A parallel digital asset can be shared across the different actors, representing this way the possession of the good, and offering the opportunity to upload documents such as invoices. The documents can be uploaded in a different section of the web app.

Background

Global supply chain network conducts an essential/strategic role in today economy. However, like in the financial sector, interactions kept growing in quantity and sophistication but centralization and old rules were not able to keep up the pace, so associations brought a monopoly full of inefficiencies leading to an obsolete system that not just put customers and suppliers at a disadvantage and risk but waste resources when are most needed and at his peak it generates global crisis. We can express gratitude for service provided in the past years, but it’s time to help them and liberate some of the main players from part of their heavy burden/load and let them focus on tasks they perform more efficiently in order to avoid future crisis. One of the main issues is around documentation and possession proof of the physical goods. A lot of organisms are involved in the process, among others governments, demanding original co-sign physical paper documents baking the product since authenticity of untampered digital copies was difficult to demonstrate. However in 2017 the United Nations Commission on International Trade Law (UNCITRAL) published the Model Law on Electronic Transferable Records (MLETR), yet to be adopted by most of the countries but if we have a look to one of the most prominent players in international trade law, UK, they are working in the Electronic Trade document Law UK which is in line with this model. Blockchain is the solution and some use cases have been successfully implementing following these guidelines. One of them is digitalization of the Bill of Landing which is a document as valuable as the goods since it travels separately from the merchandise and allows the following link in the chain to report and sign their service. It is used among banks too to unlock payments and collect the goods once unloaded in the port. An Non fungible token (NFT-ERC721) and burnable fungible tokens (ERC20) together with a decentralized application is the technology behind. Another use case is establishing a digital shadow with an NFT of the product in the blockchain. However there is a mayor flaw in all the current implementation done in this field. Once the customer is in possession of the digital and physical asset, bad actors can decide to sell things separately and a conflict of interest with the supplier can arise, particularly in the case of consumables. A month ago of writing this proposal a new White Paper was released proposing a new digital asset, SoulBound Tokens (SBT). This are non-transferable tokens that can be used as a receipt once a person has paid for a product, solving the bad actor issue and offering a membership of supplier community. At the same time a twin ERC721 can be generated in order to be transferred among the members of the supply chain allowing document upload like the bill of landing. All of this can be handled in a Decentralized Application (Dapp).

Technical Approach

In order to run this Dapp, several technologies need to merge together. As blockchain protocol, I’ll be using Ethereum as a backend. I’m aware that Polygon as a second layer is more cost efficient and suitable and if I have time I would like to explore this alternative but it’s not a priority. Smart contracts will be programmed in Solidity language and the dependencies to deploy and interact with them will be Node.js, Ganache, Truffle framework, Metamask extension and JavaScript web3 library to access it. The standards used in the smart contracts will be ERC721 for the supply chain actors and the still to be released SBT for the client receipt/membership. However as still an abstract concept yet to be materialized, I reserve the right to use different standards or none of them in order to satisfy the goal of this project. To store public immutable files in a decentralized way, I’ll be using the Interplanetary File System (IPFS) and probably Infura node to upload them. An example is an image for a SBT. For the client side application I plan to use React because it helps to create useful user interface (UI), but I do not exclude the possibility of using some sort of complementary JavaScript boiler template. And html and css. Finally I would like to deploy the webapp in AWS server and I’ll be using git and private github repository to handle the code creation.
